# F-001.2 — Org Unit Hierarchy

> Product Feature Specification. Part of EPIC-001 (Enterprise Workspace). References engineering assets; does not modify them. No code, React, or API authoring.

---

## Problem Statement

Enterprises are not flat — they are composed of divisions, departments, teams, and sub-units. Without a modeled org hierarchy, the product cannot scope intelligence to the right level (a signal about a team vs. a division), cannot roll learning up the chain (EPIC-009), and cannot present unit-level views to managers.

## Business Goal

Provide a governed, hierarchical model of organizational units (`OrgUnit` nodes) within a tenant, enabling unit-scoped intelligence, roll-ups, and manager visibility.

## User Personas

- **Tenant Administrator** — defines the org structure.
- **Org Lead / Manager** — views and manages their unit and sub-units.
- **Individual Contributor** — belongs to one or more units.

## Business Flow

1. Administrator creates a root `OrgUnit` for the tenant.
2. Child units are added under parents, forming a tree.
3. Persons (`Person`) are assigned membership in one or more units.
4. Each `OrgUnit` carries `tenantId` (mandatory) and a parent reference.
5. Downstream Epics (signals EPIC-003, cases EPIC-004, learning EPIC-009) scope and roll up by unit.

## Inputs

- Unit name, parent unit reference, unit type (division/department/team), attributes.
- Membership assignments (Person ↔ OrgUnit).

## Outputs

- `OrgUnit` nodes (id, tenantId, parent ref, type) per `graph/migrations/001_constraints.cypher`.
- Org unit tree for the tenant.
- Membership edges (Person → OrgUnit).

## Intelligence Logic

- Unit membership determines the default scoping of a person's evidence, signals, and cases.
- Roll-up logic (defined in EPIC-009) aggregates unit-level Outcomes/Learning upward along the parent chain.
- A unit's effective autonomy posture inherits from the tenant configuration (F-001.1) unless overridden.

## Related Contracts

- `graph/migrations/001_constraints.cypher` — `OrgUnit` node: `orgunit_id` unique, `orgunit_tenant` NOT NULL.
- Org hierarchy relationship is part of the canonical model; no separate `contracts/` schema yet — tracked as it emerges in `graph/`.

## Related Graph Nodes

- `OrgUnit` (id, tenantId) — primary.
- `Person` (id, tenantId) — membership.
- Implicit relationship `OrgUnit` → parent `OrgUnit` (hierarchy).

## Related APIs

- `api/` (Laravel REST) — CRUD for `OrgUnit` and membership.
- Consumed by `web/` org unit tree screen (see `development/screens/`).

## Acceptance Criteria

1. An `OrgUnit` tree can be created with arbitrary depth under a tenant.
2. Every `OrgUnit` node carries a non-null `tenantId` and cannot be created in another tenant's space.
3. Persons can be assigned to units and the assignment is tenant-scoped.
4. Unit hierarchy is queryable for downstream scoping/roll-up.
