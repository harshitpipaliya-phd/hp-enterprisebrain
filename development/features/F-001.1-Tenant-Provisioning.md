# F-001.1 — Tenant Provisioning & Isolation Boundary

> Product Feature Specification. Part of EPIC-001 (Enterprise Workspace). References engineering assets; does not modify them. No code, React, or API authoring.

---

## Problem Statement

HP Enterprise Brain serves multiple distinct enterprises on a single codebase. Without a hardened tenant boundary provisioned at the moment of onboarding, customer data, org structures, evidence, cases, and ESO executions would risk cross-tenant contamination, and every downstream Epic would have to re-solve isolation from scratch.

## Business Goal

Provide a single, governed onboarding path that creates an isolated tenant and guarantees that all subsequent graph nodes, API calls, and ESO executions are implicitly scoped to that tenant. This is the root trust boundary for the entire product.

## User Personas

- **Tenant Administrator** — initiates and configures a new tenant.
- **Platform Operator** (`infra/`) — provisions the environment that hosts the tenant.
- **Compliance Auditor** — verifies isolation guarantees.

## Business Flow

1. Tenant Administrator submits a tenant request (name, configuration, region).
2. Platform Operator / `infra/` provisions the environment boundaries.
3. System creates the tenant record and binds a `tenantId`.
4. All subsequent `Person`, `OrgUnit`, `Role`, and downstream nodes inherit the `tenantId`.
5. Isolation is enforced by `graph/migrations/001_constraints.cypher` (`tenantId NOT NULL` on every node) and the CI tenant-isolation guard (exit criterion #6, `graph/README.md`).
6. Tenant is activated and the Administrator is granted the first role.

## Inputs

- Tenant name, configuration profile, data-residency/region preference.
- Administrator identity (to receive the initial role).
- Source request (from `api/` onboarding surface).

## Outputs

- Tenant record (id, configuration) — authoritative in the system of record.
- `tenantId` bound to all created nodes.
- Activated tenant ready for EPIC-001.2–1.5 setup.

## Intelligence Logic

- On provisioning, the system tags the tenant with its configuration profile (e.g., autonomy posture, default trust ceilings) that later constrains EPIC-007/008.
- Isolation is declarative: no Cypher may run without a `tenantId` predicate; the CI guard fails any migration or query that omits it.

## Related Contracts

- `graph/migrations/001_constraints.cypher` — defines `tenantId` constraints (exit criterion #6). No field-level contract in `contracts/` governs tenancy directly yet; this feature tracks the tenant record shape as it emerges in `api/` + `graph/`.

## Related Graph Nodes

- Implicit `tenantId` attribute on **every** node: `Person`, `OrgUnit`, `Role`, `Capability`, `Evidence`, `Case`, `ESO`, `Executor`, `Outcome`, `Learning`, `Hypothesis` (all defined in `graph/migrations/001_constraints.cypher`).

## Related APIs

- `api/` (Laravel — auth, tenancy, REST, owner Vivek) — the only entry point for tenant provisioning.
- Enforced by the CI tenant-isolation workflow (`.github/workflows/tenant-isolation.yml`).

## Acceptance Criteria

1. A new tenant can be provisioned end-to-end through `api/`.
2. No Cypher query can read another tenant's `Person`, `OrgUnit`, or `Role` nodes (exit criterion #6, `graph/README.md`).
3. Every node created under the tenant carries a non-null `tenantId` (enforced by `graph/migrations/001_constraints.cypher`).
4. The tenant-isolation CI workflow passes for all migrations and queries touching this tenant.

## Development Status

**Planned.** `graph/` constraints and the tenant-isolation CI workflow exist; `api/` provisioning implementation pending.

## Dependencies

- `api/` (tenancy + auth) — implementing owner.
- `graph/migrations/001_constraints.cypher` — isolation enforcement.
- `infra/` — environment provisioning.
- No upstream product dependencies (root feature).

## Future Improvements

- Delegated/sub-tenant federation (child tenants under a parent).
- Per-tenant configuration marketplaces (autonomy postures, default trust ladders).
- Automated tenant health/isolation attestation reporting.
