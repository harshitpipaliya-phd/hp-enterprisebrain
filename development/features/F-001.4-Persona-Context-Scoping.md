# F-001.4 — Persona / Context Scoping

> Product Feature Specification. Part of EPIC-001 (Enterprise Workspace). References engineering assets; does not modify them. No code, React, or API authoring.

---

## Problem Statement

The same person operates in different contexts (a manager investigating a case vs. an executor running an ESO step). Without a persona/context scoping surface, the system cannot tailor memory, recommendations, and execution to the situation — and the per-person/per-context memory promised by the ESO contract (Block 11) has no governing boundary.

## Business Goal

Define and manage **personas/contexts** that scope a person's working surface and memory, feeding `contracts/eso/eso.schema.yaml` Block 11 `memory.scope` (enum: `per-person`, `per-context`). This makes downstream personalization (EPIC-009) and context-aware reasoning (EPIC-005) possible.

## User Personas

- **Individual Contributor** — selects/switches persona/context for a task.
- **Tenant Administrator** — defines allowable personas/templates.
- **Org Lead / Manager** — assigns role-bound contexts to team members.

## Business Flow

1. Administrator defines persona/context templates for the tenant (e.g., "Incident Investigator", "Capability Developer").
2. Persons are assigned one or more personas, optionally bound to a `Role` (F-001.3) and `OrgUnit` (F-001.2).
3. When a person enters a context, the system scopes their working surface and memory reads/writes to that context.
4. ESO execution declares `memory.scope` (Block 11); the active persona determines which memory the ESO reads/writes.
5. Memory state itself is stored in the graph (per recommended decision in `contracts/eso/eso.schema.yaml`), not in the versioned contract.

## Inputs

- Persona/context templates (name, bound role, bound unit, default memory scope).
- Person ↔ persona assignments.
- Active context selection at runtime.

## Outputs

- Persona/context definitions scoped to tenant.
- Context-scoped memory access boundary for ESO `memory` block (Block 11).
- Per-person/per-context memory references resolvable in the graph (EPIC-009).

## Intelligence Logic

- The active persona sets the memory scope (`per-person` vs `per-context`) consumed by ESO Block 11.
- Context scoping narrows the evidence/signals/cases a person sees (tenant + unit + persona filters).
- Persona templates standardize how a role operates, improving consistency of execution and learning.

## Related Contracts

- `contracts/eso/eso.schema.yaml` **Block 11 `memory`** — `scope` enum (`per-person` | `per-context`); the ESO declares what memory it reads/writes; state lives in the graph.
- `contracts/eso/eso.schema.yaml` **Block 1 `identity`** — persona may map to a `kasbaBinding` context.

## Related Graph Nodes

- `Person` (id, tenantId) — owner of personas.
- `Role` (id, tenantId) — persona binding.
- `OrgUnit` (id, tenantId) — optional scope.
- Memory state stored against `Learning`/`Outcome` ledgers (EPIC-009), tenant-scoped.

## Related APIs

- `api/` (Laravel REST) — persona/context definition and assignment.
- Consumed by `web/` role & persona management screen.

## Acceptance Criteria

1. Persona/context templates can be defined and assigned within a tenant.
2. An active persona resolves to a memory scope compatible with ESO Block 11 (`per-person`/`per-context`).
3. Persona assignment is tenant-scoped and may be bound to role/unit.
4. ESO memory reads/writes are correctly scoped by the active persona at execution time (EPIC-008/009).
