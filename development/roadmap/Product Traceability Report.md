## 3. Broken / Dangling References

### 3.1 ❌ Engineering Blueprint — DOCUMENT MISSING
The product specs (and the contract READMEs) repeatedly cite an **"Engineering Blueprint"** as an authoritative source:
- `development/roadmap/...` line 78: "Blueprint §6.3 binds per ESO".
- `development/epics/EPIC-008-...` lines 13, 62: "The Blueprint §6.3 per-ESO executor binding is incorrect".
- `contracts/README.md` line 28: "correct the Blueprint".
- Root `README.md` line 14: "The Engineering Blueprint said 'nine fields'".

**No file named `Engineering Blueprint` (or similar) exists anywhere in this repository.** It is an external/un-imported document. The product specs treat it as a reference target, but there is nothing in-repo to resolve it to. This is the single **hard broken reference** in the workspace.

**Recommendation (not implemented):** either import the Blueprint into the repo (e.g. `reference/` or a `docs/` location) or rewrite the references to point only at `contracts/eso/eso.schema.yaml` §5.2 / §5.5, which is the in-repo authority that already contradicts the Blueprint. Do not invent a new architecture.

### 3.2 ⚠️ `eso-runtime.schema.yaml` — referenced but absent
`contracts/eso/eso.schema.yaml` line 30 references `eso-runtime.schema.yaml` (the "cognitive runtime" / contract-in-motion). That file does **not** exist in `contracts/eso/`. EPIC-005/008 reference the "runtime in motion (§5.5)" concept, which depends on this missing schema. This is an engineering-layer gap, surfaced here because product specs depend on it.

### 3.3 ⚠️ `ReasoningStep` / `Task` ledgers — named, not migrated
`graph/README.md` lists `ReasoningStep` and `Task` among append-only ledgers, but `graph/migrations/` contains only `001_constraints.cypher` (which defines no `ReasoningStep` or `Task` node). EPIC-005 (ReasoningStep) and EPIC-008 (Task) trace to these ledgers; the migrations do not yet exist.
